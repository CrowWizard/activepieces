# Phase 3: Step 执行器（Worker 端）实施计划

## 概述

Phase 3 的目标是让 Worker 能够执行单个 Step，并从 Step Output Store 重建上下文。这是 Step 级别调度的核心执行层。

## 当前状态分析

### 已完成的基础设施（Phase 1 & 2）

1. **`flow_run_step` 表**：已创建，存储 step output
2. **`FlowRunStepEntity`**：已注册到 `database-connection.ts`
3. **`flowRunStepService`**：提供 `save`、`getStepOutputs`、`getStepOutput` 方法
4. **`stepOrchestrator`**：API 端编排器，处理 `onStepCompleted`、`enqueueFirstStep`、`enqueueStepResume`
5. **`ExecuteStepJobData`**：已定义在 `packages/shared/src/lib/automation/workers/job-data.ts`
6. **`WorkerJobType.EXECUTE_STEP`**：已添加到枚举
7. **`getStepOutputs` RPC**：已添加到 `WorkerToApiContract` 和 `createHandlers`
8. **`jobBroker.completeJob`**：已处理 `EXECUTE_STEP`，调用 `stepOrchestrator.onStepCompleted`

### 关键发现

1. **Worker 返回机制**：`pollAndExecute` 已支持 `SynchronousJobResult`，会将 `result.response` 传给 `completeJob`
2. **Step Output 传递**：`jobBroker.completeJob` 从 `input.response` 提取 `StepOutput`，传给 `stepOrchestrator.onStepCompleted`
3. **FlowExecutorContext 重建**：需要新增 `fromStepOutputs` 静态方法
4. **执行模式**：可复用现有 Sandbox 机制，或使用 `UNSANDBOXED` 模式直接在 Worker 进程内执行

## 实施任务

### 任务 3.1：新增 `executeStepJob` handler

**文件**：`packages/server/worker/src/lib/execute/jobs/execute-step.ts`

**实现要点**：

```typescript
import {
    ActivepiecesError,
    EngineOperationType,
    EngineResponseStatus,
    ErrorCode,
    ExecuteStepJobData,
    ExecutionType,
    FlowActionType,
    FlowRunStatus,
    FlowVersion,
    GenericStepOutput,
    isNil,
    ResumeReason,
    StepOutput,
    StepOutputStatus,
    tryCatch,
    WorkerJobType,
} from '@activepieces/shared'
import { flowCache } from '../../cache/flow/flow-cache'
import { workerSettings } from '../../config/worker-settings'
import { JobContext, JobHandler, JobResultKind, SynchronousJobResult } from '../types'
import { provisionFlowPieces } from '../utils/flow-helpers'

export const executeStepJob: JobHandler<ExecuteStepJobData, SynchronousJobResult> = {
    jobType: WorkerJobType.EXECUTE_STEP,
    async execute(ctx: JobContext, data: ExecuteStepJobData): Promise<SynchronousJobResult> {
        const timeoutInSeconds = workerSettings.getSettings().FLOW_TIMEOUT_SECONDS

        // 1. 获取 FlowVersion
        const flowVersion = await flowCache(ctx.log, ctx.apiClient).getVersion({ flowVersionId: data.flowVersionId })
        if (isNil(flowVersion)) {
            ctx.log.info({ flowVersionId: data.flowVersionId }, 'Flow version not found')
            return {
                kind: JobResultKind.SYNCHRONOUS,
                status: EngineResponseStatus.INTERNAL_ERROR,
                response: createFailedStepOutput(data.stepName, 'Flow version not found'),
                errorMessage: 'Flow version not found',
            }
        }

        // 2. Provision pieces (只针对当前 step)
        const { data: provisioned, error: provisionError } = await tryCatch(() =>
            provisionFlowPieces({
                flowVersion,
                platformId: data.platformId,
                flowId: flowVersion.flowId,
                projectId: data.projectId,
                log: ctx.log,
                apiClient: ctx.apiClient,
            })
        )
        if (provisionError || !provisioned) {
            return {
                kind: JobResultKind.SYNCHRONOUS,
                status: EngineResponseStatus.INTERNAL_ERROR,
                response: createFailedStepOutput(data.stepName, 'Failed to provision pieces'),
                errorMessage: provisionError?.message ?? 'Failed to provision pieces',
            }
        }

        // 3. 从 API 读取前序 steps output (Memoization)
        const stepOutputs = await ctx.apiClient.getStepOutputs({
            flowRunId: data.flowRunId,
            projectId: data.projectId,
        })

        // 4. 执行 step
        const sandbox = ctx.sandboxManager.acquire({ log: ctx.log, apiClient: ctx.apiClient })
        try {
            await sandbox.start({
                flowVersionId: flowVersion.id,
                platformId: data.platformId,
                mounts: [],
            })

            const operation = buildStepOperation(ctx, data, flowVersion, stepOutputs, timeoutInSeconds)
            const result = await sandbox.execute(
                EngineOperationType.EXECUTE_STEP,  // 新增
                operation,
                { timeoutInSeconds },
            )

            if (result.status !== EngineResponseStatus.OK) {
                return {
                    kind: JobResultKind.SYNCHRONOUS,
                    status: EngineResponseStatus.OK,  // 即使 step 失败也返回 OK，由编排者处理
                    response: createFailedStepOutput(data.stepName, result.error ?? 'Step execution failed'),
                }
            }

            return {
                kind: JobResultKind.SYNCHRONOUS,
                status: EngineResponseStatus.OK,
                response: result.response as StepOutput,
            }
        }
        catch (e) {
            await ctx.sandboxManager.invalidate(ctx.log)
            // ... 错误处理
        }
        finally {
            await ctx.sandboxManager.release(ctx.log)
        }
    },
}
```

### 任务 3.3：新增 `stepOperation` (Engine 端)

**文件**：`packages/server/engine/src/lib/operations/step.operation.ts`

**实现要点**：

```typescript
import { EngineResponse, EngineResponseStatus, ExecuteStepOperation, ExecutionType, FlowActionType, FlowRunStatus, GenericStepOutput, isNil, ResumeReason, StepOutputStatus } from '@activepieces/shared'
import { flowExecutor } from '../handler/flow-executor'
import { EngineConstants } from '../handler/context/engine-constants'
import { FlowExecutorContext } from '../handler/context/flow-execution-context'
import { flowStructureUtil } from '@activepieces/shared'

export const stepOperation = {
    async execute(operation: ExecuteStepOperation): Promise<EngineResponse<StepOutput>> {
        const { flowVersion, stepName, stepOutputs, executionType, resumePayload, resumeReason } = operation

        // 1. 从 stepOutputs 重建 FlowExecutorContext
        const executionState = FlowExecutorContext.fromStepOutputs({
            steps: stepOutputs,
            engineApi: {
                engineToken: operation.engineToken,
                internalApiUrl: operation.internalApiUrl,
            },
        })

        // 2. 找到当前 step
        const step = flowStructureUtil.getActionOrThrow(stepName, flowVersion.trigger)

        // 3. 构建 EngineConstants
        const constants = EngineConstants.fromExecuteStepInput(operation)

        // 4. 执行 step
        const result = await flowExecutor.execute({
            action: step,
            executionState,
            constants,
        })

        // 5. 提取 step output
        const stepOutput = result.getStepOutput(stepName)
        if (isNil(stepOutput)) {
            return {
                status: EngineResponseStatus.INTERNAL_ERROR,
                response: GenericStepOutput.create({
                    type: step.type,
                    status: StepOutputStatus.FAILED,
                    input: {},
                }).setErrorMessage('Step output not found after execution'),
            }
        }

        return {
            status: EngineResponseStatus.OK,
            response: stepOutput,
        }
    },
}
```

static fromStepOutputs(params: {
    steps: Record<string, StepOutput>
    engineApi?: EngineApiConfig
    slicingEnabled?: boolean

        steps: params.steps,

        currentPath: StepExecutionPath.empty(),

## 关键设计决策

### 1. Step Output 重建策略

- **问题**：`FlowExecutorContext.steps` 包含所有 step output，包括 Loop 内部的迭代
- **方案**：`fromStepOutputs` 直接使用从 API 获取的 `Record<string, StepOutput>`，Loop 迭代数据已通过 `executionJournal` 的结构存储在 `LoopStepOutput.output.iterations` 中

### 2. Router/Loop 处理

- **当前阶段**：Phase 3 只实现单个 Step 的执行，Router 和 Loop 的编排逻辑在 Phase 2 的 `stepOrchestrator` 中处理
- **Loop 内部 Step**：Loop 内部的 step 仍作为一个整体执行（由 `loopExecutor` 内部调用 `flowExecutor.execute`），不拆分为独立 Job

### 3. 执行模式选择

- **默认**：使用现有 Sandbox 机制，每个 step 启动独立 sandbox
- **优化路径**：`UNSANDBOXED` 模式下可直接在 Worker 进程内调用 Engine 函数，消除进程启动开销

### 4. 错误处理

- **Step 失败**：返回 `SynchronousJobResult`，`status: OK`，`response` 包含失败的 `StepOutput`
- **编排者处理**：`stepOrchestrator.onStepCompleted` 检查 `stepOutput.status === FAILED`，更新 Flow Run 状态

## 测试策略

### 单元测试

1. **`FlowExecutorContext.fromStepOutputs`**：验证从 `Record<string, StepOutput>` 重建的 context 与原始 context 一致
2. **`stepOperation.execute`**：验证单个 step 执行的正确性

### 集成测试

1. **线性 Flow**：3 个 step 的线性 flow，验证每个 step 独立执行
2. **Router Flow**：包含 Router 的 flow，验证分支选择正确
3. **Loop Flow**：包含 Loop 的 flow，验证迭代执行正确
4. **Resume Flow**：Waitpoint 暂停后恢复，验证 `RESUME` 执行类型

## 风险与缓解

| 风险 | 严重度 | 缓解措施 |
|---|---|---|
| `FlowExecutorContext` 重建不完整 | 高 | 逐字段对比测试，确保 `steps`、`verdict`、`currentPath` 等字段正确 |
| Loop 迭代数据丢失 | 高 | 确保 `LoopStepOutput.output.iterations` 正确序列化/反序列化 |
| Step 执行超时 | 中 | 复用现有 `FLOW_TIMEOUT_SECONDS`，后续可按 step 级别配置 |
| Sandbox 启动开销 | 中 | 优先使用 `UNSANDBOXED` 模式测试 |
