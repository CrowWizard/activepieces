# 图片背景生成 + 细节优化工作流方案

## 概述

构建一个 Activepieces 自动化流程，接收外部图片，经过 **去底→生成背景→细节优化** 三步处理，最终输出成品图，并以 Webhook API 形式暴露给其他应用调用。

## 架构关系

```
外部应用 ──POST──→ Webhook 触发器 (sync)
                      │
                      ▼
              ┌──────────────────────────┐
              │ Step 1: Photoroom        │
              │ Remove Background        │  → { fileName, url }
              └──────────┬───────────────┘
                         │  (内部文件 URL)
                         ▼
              ┌──────────────────────────────────┐
              │ Step 2: Tongyi Generate BG       │
              │ ① 下载 Step1 内部 URL → base64    │
              │ ② 调 DashScope 异步任务 API       │
              │ ③ 轮询(2s间隔,上限120s) → 结果URL │
              │ ④ 下载结果图 → files.write()      │  → { fileName, url }
              └──────────┬───────────────────────┘
                         │  (内部文件 URL)
                         ▼
              ┌──────────────────────────────────┐
              │ Step 3: Tongyi Image Pro Edit    │
              │ ① 下载 Step2 内部 URL → base64    │
              │ ② 调 DashScope 异步任务 API       │
              │ ③ 轮询(2s间隔,上限120s) → 结果URL │
              │ ④ 下载结果图 → files.write()      │  → { fileName, url }
              └──────────┬───────────────────────┘
                         │
                         ▼
              ┌──────────────────────────┐
              │ 返回: {                   │
              │   originalUrl,           │
              │   transparentUrl,        │
              │   backgroundUrl,         │
              │   finalUrl               │
              │ }                        │
              └──────────────────────────┘
```

## 关键技术约束

### 1. 内部文件 URL 不可达问题

Photoroom 输出的 `url` 是 Activepieces 内部文件服务地址（类似 `/api/v1/files/xxx`），DashScope API 无法访问。

**解决**：Tongyi piece 的每个 action 必须先通过 `httpClient` 下载上游步骤的内部文件 URL，转为 base64，再传给 DashScope。

### 2. DashScope 异步任务模式

`wanx-background-generation-v3` 和 `wan2.7-image-pro` 均为异步任务模式：

- `POST` 提交任务 → 返回 `{ output: { task_id: "xxx" } }`
- `GET /api/v1/tasks/{task_id}` 轮询，状态 `SUCCEEDED` 时取 `results.output.url`
- 轮询策略：2s 间隔，指数退避（2s→4s→8s），上限 120s
- 超时返回友好错误消息，不抛未捕获异常

### 3. 同步 Webhook 超时风险

三个步骤串行总耗时可能 30-180s。需确认 Fastify `requestTimeout` ≥ 180s。

**备选方案**：改用异步 Webhook 触发器，立即返回 `{ flowRunId: "xxx" }`，调用方轮询 `GET /api/v1/flow-runs/{id}` 获取结果。

### 4. 文字叠加能力边界

DashScope 图像生成 API **不原生支持精确文字叠加**，只能通过 prompt 描述（如"在右上角添加金色文字'新品上市'"），效果不可控。

**建议**：文字叠加拆为独立 action，用 `sharp` 库在服务端直接绘制文字到图片上；或交给调用方自行处理。Flow 中 `imageProEdit` 只负责局部重绘（inpaint）。

## 需要构建的组件

### 1. 新建 Piece: `@activepieces/piece-tongyi`

**位置**: `packages/pieces/community/tongyi/`

**原因**：
- 开发期放 community 目录，走标准构建链路（turbo build、tsconfig 路径映射、lint）
- `packages/pieces/custom/` 当前无实际 piece 先例，且 custom piece 机制是运行时动态上传 archive，不适合开发期迭代
- 生产部署时可通过 `npx turbo build --filter=@activepieces/piece-tongyi` + 打包 archive + API 上传到平台 custom piece

**Action 定义**:

| Action 名称 | 对应的 DashScope API | 功能 |
|---|---|---|
| `generateBackground` | `POST /api/v1/services/aigc/background-generation/generation` | 接收透明 PNG(base64) + 背景描述 prompt，异步生成带背景图片 |
| `imageProEdit` | `POST /api/v1/services/aigc/image2image/image-synthesis` | 接收图片(base64) + inpaint mask + 重绘 prompt，异步输出优化后图片 |

**认证方式**: `PieceAuth.CustomAuth({ props: { apiKey: PieceAuth.SecretText() } })` — 阿里云 DashScope API Key（Bearer Token）

**主要依赖**: `@activepieces/pieces-common` 的 `httpClient` 发送 HTTP 请求

**client.ts 核心逻辑**:

```
submitAndWait(params):
  1. 构造 DashScope payload（model, input, parameters）
  2. POST 提交异步任务 → 取 task_id
  3. 轮询 GET /api/v1/tasks/{task_id}：
     - interval: 2s, 指数退避, 上限 120s
     - PENDING/RUNNING → 继续等待
     - SUCCEEDED → 下载结果图 URL → files.write() → 返回内部 URL
     - FAILED → 抛 ActivepiecesError
     - 超时 → 抛 ActivepiecesError(ENGINE_OPERATION_FAILURE)
```

**文件清单**:
```
packages/pieces/community/tongyi/
├── package.json
├── tsconfig.json
├── tsconfig.lib.json
├── .eslintrc.json
└── src/
    ├── index.ts              # Piece 入口，注册 auth + actions
    └── lib/
        ├── common/
        │   └── client.ts     # DashScope API 客户端（下载→base64→提交→轮询→下载结果→写入）
        └── actions/
            ├── generate-background.ts   # wanx-background-generation-v3
            └── image-pro-edit.ts        # wan2.7-image-pro (局部重绘)
```

**注册**: 在 `tsconfig.base.json` 添加路径映射：
```json
"@activepieces/piece-tongyi": ["packages/pieces/community/tongyi/src/index.ts"]
```

### 2. 创建 Activepieces Flow

**触发器**: **Webhook 触发器**（同步响应模式，`/api/v1/webhooks/:flowId/sync`）

**Flow steps**:

| Step | Piece | Action | 输入 | 输出 |
|---|---|---|---|---|
| 1 | Photoroom | Remove Background | 原始图片文件 | `{ fileName, url }` 透明 PNG |
| 2 | Tongyi | Generate Background | Step1.url + backgroundPrompt | `{ fileName, url }` 带背景图 |
| 3 | Tongyi | Image Pro Edit | Step2.url + editPrompt + editMask(可选) | `{ fileName, url }` 成品图 |
| 4 | (返回) | - | 汇总各步骤结果 | 返回给调用方 |

**Flow 输入参数**（Webhook Body）:
```json
{
  "imageUrl": "https://...original.png",
  "backgroundPrompt": "现代简约白色客厅，自然光，高清",
  "editPrompt": "优化光影效果，增强质感",
  "editMask": "..."  // 可选，局部重绘区域（base64 PNG mask）
}
```

**返回值**:
```json
{
  "originalUrl": "https://...original.png",
  "transparentUrl": "内部文件URL/透明PNG",
  "backgroundUrl": "内部文件URL/带背景图",
  "finalUrl": "内部文件URL/最终成品图"
}
```

### 3. 暴露为外部 API

Activepieces 的 Webhook 触发器天然提供 HTTP 端点：
- **同步模式**: `POST /api/v1/webhooks/{flowId}/sync` — 等待流程执行完毕后返回结果
- 外部应用只需 POST 图片 URL + 参数到该地址即可使用
- 不需要额外开发 API 网关层

**注意事项**：
- 确认 Fastify `requestTimeout` 配置 ≥ 180s（三步串行最大耗时）
- 若调用方无法等待，改用异步 Webhook + 轮询 `GET /api/v1/flow-runs/{id}`

### 4. Photoroom API 密钥

步骤 1 使用现有的 `@activepieces/piece-photoroom`，需要在 Activepieces 中配置 Photoroom API Key。

## 实施步骤

### Phase 1: 创建 Tongyi Piece

1. 在 `packages/pieces/community/tongyi/` 下搭建基础文件结构（参照 photoroom piece 的 package.json / tsconfig 结构）
2. 实现 `common/client.ts` — DashScope 异步任务客户端：
   - 下载内部文件 URL → base64 转换
   - 提交异步任务
   - 指数退避轮询（2s→4s→8s，上限 120s）
   - 下载结果图 URL → `files.write()` → 返回内部 URL
   - 超时 / 失败友好错误处理
3. 实现 `actions/generate-background.ts` — 调用 `wanx-background-generation-v3`
4. 实现 `actions/image-pro-edit.ts` — 调用 `wan2.7-image-pro`（局部重绘）
5. 实现 `src/index.ts` — 注册 auth + actions
6. 注册到 `tsconfig.base.json` 添加路径映射
7. 运行 `bun install && npx turbo build --filter=@activepieces/piece-tongyi` 验证编译
8. `npm run lint-dev` 验证代码规范

### Phase 2: 创建 Flow

1. 启动 dev 环境：`npm start`
2. 确认 Fastify `requestTimeout` 配置，必要时调大至 180s+
3. 在 Flow Builder 中：
   - 添加 **Webhook 触发器**（同步模式）
   - 添加 **Photoroom: Remove Background**
   - 添加 **Tongyi: Generate Background**
   - 添加 **Tongyi: Image Pro Edit**
4. 配置各步骤参数映射（Step N 的输出引用 Step N-1 的结果）
5. 启用 Flow，获取 Webhook URL

### Phase 3: 测试

1. 用 curl / Postman POST 到 Webhook URL 端到端测试
2. 验证三步串联：去底 → 生成背景 → 局部重绘
3. 验证超时场景（DashScope 慢响应）的错误处理
4. 验证返回值包含各步骤 URL

## 不涉及的内容

- 无需新增数据库 Entity / Migration（流程定义用现有 Flow 表）
- 无需新增 API 端点（Webhook 触发器自带 HTTP 入口）
- 无需改动 worker / engine（piece 执行走现有流程）
- 无需新增 plan flag（个人工作流，非 EE 功能）
- 文字叠加不在本 Flow 中实现（DashScope 无原生支持，建议调用方自行处理或后续扩展独立 action）

## 风险矩阵

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| DashScope 异步任务超时 | 中 | 调用方 504 | 轮询上限 120s + 超时友好错误消息 + 确认 requestTimeout |
| 文字叠加效果不可控 | 高 | 输出质量差 | 本期不纳入 Flow，由调用方自行处理 |
| 内部文件 URL 不可达 | 高 | Step 2/3 全部失败 | client.ts 必须先下载转 base64（已在设计中） |
| Webhook 同步超时 | 中 | 调用方收不到结果 | 调大 requestTimeout 或改异步模式 |
| DashScope API 变更 | 低 | piece 失效 | client.ts 做好错误码映射和版本化 |
