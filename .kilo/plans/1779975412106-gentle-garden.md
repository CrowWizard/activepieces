# Plan: postgres-task-queue Piece

## 目标

在 `packages/pieces/community/postgres-task-queue/` 下创建一个新 piece，实现基于 PostgreSQL 的任务队列方案：

1. **FOR UPDATE SKIP LOCKED**：多 worker 安全抢锁（核心能力）
2. **POLLING + 可配置轮询间隔**：通过 `ctx.setSchedule()` 支持用户自定义轮询频率
3. **兜底轮询**：防通知丢失

## 架构分析

### Activepieces Trigger 模型约束

Activepieces 的 trigger 运行模型是 **短生命周期** 的：
- `POLLING` trigger：由 worker 定期调用 `run()`，每次执行是独立的短连接
- `WEBHOOK` trigger：外部 HTTP POST 到 `context.webhookUrl`，由 Activepieces 路由到 `run()`
- `APP_WEBHOOK` trigger：平台级 webhook，类似 WEBHOOK

**关键限制**：Activepieces **没有**内置的"长连接 trigger"类型。每次 `run()` 调用都是独立的、短生命周期的。LISTEN/NOTIFY 需要 PG 长连接持续监听，这与 Activepieces trigger 模型不匹配。

### 解决方案

采用 **POLLING + 可配置轮询间隔 + FOR UPDATE SKIP LOCKED** 的混合模式：

1. **Trigger 类型**：使用 `TriggerStrategy.POLLING`
2. **轮询间隔**：在 `onEnable` 中调用 `ctx.setSchedule()` 设置用户指定的 cron 表达式，覆盖平台默认的 5 分钟间隔
3. **FOR UPDATE SKIP LOCKED**：在 `run()` 中使用 `SELECT ... FOR UPDATE SKIP LOCKED` 获取并锁定任务，防止重复消费
4. **去重策略**：不使用 `DedupeStrategy.LAST_ITEM`（与抢锁消费模式冲突），改为自定义去重逻辑 + `_dedupe_key` 服务端去重

### 为什么不使用 DedupeStrategy.LAST_ITEM

`DedupeStrategy.LAST_ITEM` 的设计假设是：
- 数据源是 append-only 的（新行不断追加，旧行不变）
- `items()` 返回 newest-to-oldest 的列表
- 通过 `lastItem` 锚点截取新项

但任务队列模式是 **消费型** 的：
- 任务被 claim 后 status 变为 `claimed`，后续 poll 不会再获取到
- 不存在"同一行在多次 poll 中出现"的场景（FOR UPDATE SKIP LOCKED 已保证）
- `LAST_ITEM` 的锚点逻辑在消费型场景下无意义——每次 poll 获取的都是全新的 pending 任务

因此采用 **自定义 run() 逻辑**，不依赖 `pollingHelper`，直接在 `run()` 中执行 FOR UPDATE SKIP LOCKED 查询，并使用 `_dedupe_key` 作为并发安全网。

### 为什么移除 LISTEN/NOTIFY

在 Activepieces piece 框架内：
- `run()` 是短生命周期调用，无法维持 PG 长连接来 LISTEN
- `setup-notify-trigger` action 只能在 PG 侧创建触发器，但 piece 内无法利用通知
- 用户若需低延迟，应使用 `ctx.setSchedule()` 缩短轮询间隔（最低 1 分钟），或通过外部进程 LISTEN 后调用 Activepieces webhook trigger

移除 `setup-notify-trigger` action 可避免给用户造成"配置了通知就能即时触发"的误解。

### 方案总览

| 机制 | 实现方式 | 优先级 |
|------|----------|--------|
| FOR UPDATE SKIP LOCKED | `run()` 中执行 `SELECT ... FOR UPDATE SKIP LOCKED` 获取并锁定任务 | 核心 |
| 可配置轮询间隔 | `onEnable` 中调用 `ctx.setSchedule()` 设置用户指定的 cron | 核心 |
| 兜底轮询 | Activepieces 内置 POLLING 机制 | 核心 |
| `_dedupe_key` 服务端去重 | 每个 claimed 任务使用 `id` 作为去重 key，防止并发 poll 重复触发 | 安全网 |

**核心价值**：FOR UPDATE SKIP LOCKED 保证了多 worker 场景下的任务安全获取，这是现有 postgres piece 的 `new-row` trigger 不具备的。可配置轮询间隔让用户在延迟和数据库负载之间自主权衡。

## 约定表结构

piece 约定一个任务表结构（用户可自定义表名）：

```sql
CREATE TABLE {table_name} (
  id            BIGSERIAL PRIMARY KEY,
  payload       JSONB NOT NULL DEFAULT '{}',
  status        TEXT NOT NULL DEFAULT 'pending',   -- pending / claimed / done / failed
  claimed_by    TEXT,                               -- worker 标识
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  claimed_at    TIMESTAMPTZ,
  completed_at  TIMESTAMPTZ
);
```

piece 提供 `setup-task-table` action 自动建表。

## 详细实现

### Auth

与现有 postgres piece 相同的 `PieceAuth.CustomAuth`（host/port/user/password/database/ssl 配置），但独立定义以避免依赖。

Auth props：

| Property | Type | Required | Default | Description |
|---|---|---|---|---|
| `host` | `ShortText` | yes | - | Hostname of the PostgreSQL server |
| `port` | `Number` | yes | `5432` | Port of the PostgreSQL server |
| `user` | `ShortText` | yes | - | User to authenticate as |
| `password` | `SecretText` | yes | - | Password for authentication |
| `database` | `ShortText` | yes | - | Database name to connect to |
| `enable_ssl` | `Checkbox` | yes | `true` | Connect over SSL |
| `reject_unauthorized` | `Checkbox` | yes | `false` | Verify server certificate against trusted CAs |
| `certificate` | `LongText` | no | `''` | CA certificate for verification |

Validation：尝试连接 PG 并立即断开，成功返回 `{ valid: true }`，失败返回 `{ valid: false, error }`。

### Actions

#### 1. `setup-task-table`
- 输入：`table_name`（默认 `tasks`）
- 执行：`CREATE TABLE IF NOT EXISTS` 创建约定表结构
- 输出：`{ created: true, table_name }`
- SQL 使用 `pg-format` 的 `%I` 进行标识符安全引用

#### 2. `claim-task`
- 输入：`table_name`、`worker_id`（可选，默认自动生成 UUID）、`batch_size`（可选，默认 1）
- 执行：
  ```sql
  UPDATE %I SET
    status = $1,
    claimed_by = $2,
    claimed_at = now()
  WHERE id IN (
    SELECT id FROM %I
    WHERE status = 'pending'
    ORDER BY id ASC
    LIMIT $3
    FOR UPDATE SKIP LOCKED
  )
  RETURNING *
  ```
  其中 `%I` 使用 `pg-format` 替换为 `table_name`
- 输出：claimed 任务数组

#### 3. `complete-task`
- 输入：`table_name`、`task_id`
- 执行：`UPDATE %I SET status = 'done', completed_at = now() WHERE id = $1`
- 输出：更新后的行

#### 4. `fail-task`
- 输入：`table_name`、`task_id`、`error_message`（可选）
- 执行：`UPDATE %I SET status = 'failed', completed_at = now() WHERE id = $1`
- 输出：更新后的行

### Trigger: `new-task`

- 类型：`TriggerStrategy.POLLING`
- **不使用** `DedupeStrategy`（与消费型模式冲突），自定义 `run()` / `test()` / `onEnable()` / `onDisable()`
- 输入 props：
  - `table_name`（默认 `tasks`）
  - `worker_id`（worker 标识，默认自动生成 UUID）
  - `batch_size`（每次获取数量，默认 1）
  - `poll_interval_minutes`（轮询间隔分钟数，默认 1，范围 1-59）
- `onEnable()` 逻辑：
  1. 调用 `ctx.setSchedule({ cronExpression: '*/' + pollIntervalMinutes + ' * * * *', timezone: 'UTC' })` 设置轮询间隔
  2. 不需要初始化 store（消费型模式无需锚点）
- `onDisable()` 逻辑：无操作
- `run()` 逻辑：
  1. 创建 PG 连接
  2. 执行 FOR UPDATE SKIP LOCKED 查询获取 pending 任务：
     ```sql
     UPDATE %I SET
       status = $1,
       claimed_by = $2,
       claimed_at = now()
     WHERE id IN (
       SELECT id FROM %I
       WHERE status = 'pending'
       ORDER BY id ASC
       LIMIT $3
       FOR UPDATE SKIP LOCKED
     )
     RETURNING *
     ```
  3. 将每个 claimed 任务映射为 trigger 输出，附加 `_dedupe_key` 属性（值为 `String(task.id)`）作为服务端去重安全网
  4. 关闭 PG 连接（在 `finally` 块中）
  5. 返回 claimed 任务数组（每个任务触发一个独立的 flow run）
- `test()` 逻辑：
  1. 查询最近 5 条 pending 任务（不 claim，仅 SELECT）
  2. 返回查询结果作为样本数据
- `sampleData`：提供一个静态的示例任务对象

### `_dedupe_key` 服务端去重

每个 `run()` 返回的 trigger 输出项包含 `_dedupe_key: String(task.id)` 属性。服务端去重服务（`dedupe-service.ts`）会在 30 秒窗口内检查此 key，如果同一 `flowVersionId` 下有重复 key，则丢弃该 payload。这是防止并发 poll 执行导致重复触发的安全网。

`_dedupe_key` 属性在 payload 传递给 flow 之前会被服务端自动剥离，不会出现在 flow 的 trigger 输出中。

## 与现有 postgres piece 的关系

- **完全独立**的 piece，不依赖也不修改现有 postgres piece
- **互补关系**：
  - 现有 postgres piece 的 `new-row` trigger：适用于 append-only 场景（新行不断追加，旧行不变），使用 `DedupeStrategy.LAST_ITEM` 去重
  - 本 piece 的 `new-task` trigger：适用于消费型任务队列场景（任务被 claim 后状态变更），使用 FOR UPDATE SKIP LOCKED 抢锁

## 风险与注意事项

1. **连接管理**：每次 `run()` 需新建 PG 连接（piece 是无状态的），高频率轮询下需注意连接池压力。建议在文档中说明轮询间隔与连接压力的权衡
2. **FOR UPDATE SKIP LOCKED 需要 PG 9.5+**：应在文档中说明
3. **任务完成确认**：piece trigger claim 了任务后，如果 flow 执行失败，任务状态需用户自行处理（可提供 `fail-task` action 在 error branch 中调用）
4. **轮询间隔下限**：cron 表达式最小粒度为 1 分钟，无法实现秒级轮询。如需更低延迟，建议用户通过外部进程 LISTEN 后调用 Activepieces webhook trigger
5. **`_dedupe_key` 窗口**：服务端去重窗口为 30 秒，超过 30 秒的并发 poll 不会被去重。但 FOR UPDATE SKIP LOCKED 已在数据库层面保证不会重复 claim，`_dedupe_key` 仅作为额外安全网
6. **表名 SQL 注入**：所有表名使用 `pg-format` 的 `%I` 进行标识符引用，防止 SQL 注入
7. **与现有 postgres piece 的关系**：完全独立的 piece，不依赖也不修改现有 postgres piece
