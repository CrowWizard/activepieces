# Plan: Step 级别调度 — 参照 Inngest 实现

## 项目类型判断

- **类型**：已有项目（Activepieces）
- **目标**：架构改造 — 将 BullMQ 调度粒度从 Flow Run 级别改为 Step 级别，支持同一 Workflow 的不同 Step 在不同 Worker 上执行
- **参照模型**：Inngest 的 Memoization + Step Output Store 模式

## Discovery 结果摘要

### 当前架构关键事实

1. **调度粒度**：`WorkerJobType.EXECUTE_FLOW` — 整个 Flow Run 是一个 BullMQ Job
2. **执行模型**：Engine 的 `flowExecutor.execute` 用 `while` 循环同步执行所有 step，`FlowExecutorContext` 在内存中流转
3. **队列路由**：按 `platformId` 路由到 `WorkerGroup` 对应队列（EE 功能），不是按 step 路由
4. **暂停/恢复**：`waitpoint` 机制 — step 设置 `createWaitpoint` → engine 暂停 → API 将 Flow Run 状态设为 PAUSED → 恢复时重新入队 `EXECUTE_FLOW`（`ExecutionType.RESUME`）
5. **状态持久化**：`FlowExecutorContext.steps` 通过 `flowRunProgressReporter.backup()` 序列化后上传到文件服务（zstd 压缩），恢复时通过 `fetchExecutionStateFromLogs` 下载反序列化
6. **Sandbox**：Engine 在独立子进程中运行（fork/isolate），Worker 通过 Socket.IO RPC 调用。`ExecutionMode.UNSANDBOXED` 可跳过进程隔离
7. **StepOutput 结构**：`GenericStepOutput { type, status, input, output, outputType, duration, errorMessage }` — 纯数据对象，可 JSON 序列化
8. **ExecutionState 结构**：`{ steps: Record<string, StepOutput>, tags: string[] }` — 已有 zod schema
9. **Flow 结构遍历**：`flowStructureUtil` 可遍历 FlowVersion JSON 的 trigger → action 链，支持 `getAllSteps`、`getActionOrThrow`、`getAllNextActionsWithoutChildren`
10. **FlowRun 表**：`flow_run` 表没有 `steps` 列（steps 存在文件中），但有 `status`、`logsFileId`、`stepsCount`、`failedStep` 等字段

### 核心约束

- Workflow 是 JSON 配置（`FlowVersion`），不是代码 — 无法直接使用 Temporal 的 re-play 代码模型
- `GenericStepOutput` 是不可变值对象，天然可序列化 — 这是实现 Step Memoization 的基础
- Router 和 Loop 使 step 间的执行路径动态化，编排者必须在运行时才能确定下一步
- 当前 `flow_run.steps` 数据存在文件中（通过 logsFileId），不在 PostgreSQL 中 — 需要新增存储层或改造现有存储
- 多租户：所有查询必须按 `projectId` / `platformId` 隔离

### 参照 Inngest 的关键映射

| Inngest 概念 | Activepieces 对应 |
|---|---|
| Inngest Server (Orchestrator) | API Server |
| Function Runner (HTTP callback) | Worker (BullMQ poll) |
| `step.run()` | `pieceExecutor.handle()` / `codeExecutor.handle()` |
| Step Output Store | `flow_run_steps` 表（需新建）或现有文件存储 |
| Memoization | 从 Step Output Store 读取已完成 step 的结果 |
| Event | `waitpoint` 的 resume signal |

## Writing-Plans

### 设计目标

1. 每个 Step 可以作为独立的 BullMQ Job 入队，路由到不同 Worker Group
2. API Server 成为编排者（类似 Inngest Server），决定下一步执行哪个 step
3. Worker 只负责执行单个 Step，执行完返回结果
4. 已完成 Step 的 Output 持久化到 Step Output Store，后续 Step 通过 Memoization 读取
5. 与现有 Flow Run 级别调度共存，通过 Feature Flag 切换

### 执行流程

```
┌──────────┐     ┌───────────┐     ┌──────────┐     ┌───────────┐
│ Trigger   │────▶│ Step 1    │────▶│ Step 2   │────▶│ Step 3    │
│ Execute   │     │ (Worker A)│     │ (Worker B)│     │ (Worker A)│
└──────────┘     └───────────┘     └──────────┘     └───────────┘

1. Trigger 执行完成
   → API: 保存 trigger output 到 Step Output Store
   → API: 遍历 FlowVersion, 找到 firstAction
   → API: 根据 step 的 workerGroup 配置决定目标队列
   → API: 入队 EXECUTE_STEP Job (stepName=step1, queue=group-a)

2. Worker A 拉到 Step 1 Job
   → Worker: 从 Step Output Store 读取前序 steps output (Memoization)
   → Worker: 重建 FlowExecutorContext
   → Worker: 执行 step1 (pieceExecutor / codeExecutor / routerExecutor)
   → Worker: 返回 step1 output 给 API

3. API 收到 Step 1 完成事件
   → API: 保存 step1 output 到 Step Output Store
   → API: 遍历 FlowVersion, 找到 step1.nextAction
   → API: 根据 step2 的 workerGroup 配置决定目标队列
   → API: 入队 EXECUTE_STEP Job (stepName=step2, queue=group-b)

4. Worker B 拉到 Step 2 Job
   → Worker: 从 Step Output Store 读取 step1 output (Memoization)
   → Worker: 重建 FlowExecutorContext
   → Worker: 执行 step2
   → Worker: 返回 step2 output 给 API

5. 以此类推, 直到没有 nextAction → Flow Run 完成
```

#### Phase 1: Step Output Store（基础设施）

**目标**：建立 Step 级别的输出持久化能力，为后续 Memoization 打基础。

##### 1.1 新建 `flow_run_step` 表

```sql
CREATE TABLE flow_run_step (
  id            VARCHAR(21) PRIMARY KEY,    -- ApId
  "flowRunId"   VARCHAR(21) NOT NULL,
  "projectId"   VARCHAR(21) NOT NULL,
  step_name     VARCHAR(255) NOT NULL,
  step_type     VARCHAR(50) NOT NULL,       -- PIECE / CODE / ROUTER / LOOP_ON_ITEMS / TRIGGER
  status        VARCHAR(20) NOT NULL,       -- SUCCEEDED / FAILED / PAUSED / RUNNING
  input         JSONB,                      -- 解析后的输入 (censoredInput)
  output        JSONB,                      -- step 执行结果
  duration      INTEGER,                    -- 执行时长 ms
  error_message JSONB,                      -- 失败信息
  path          JSONB DEFAULT '[]',         -- loop 路径: [["loopStep", 0]]
  queue_name    VARCHAR(255),               -- 执行该 step 的队列名
  created       TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated       TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_flow_run_step_run_id ON flow_run_step ("flowRunId");
CREATE INDEX idx_flow_run_step_run_step ON flow_run_step ("flowRunId", step_name);
```

**设计决策**：
- `output` 字段使用 JSONB 而非文件存储 — 方便 Worker 快速读取前序 step 结果（Memoization），避免文件下载开销
- `path` 字段记录 loop 上下文 — 复用 `executionJournal` 的路径模型
- `queue_name` 记录该 step 被路由到了哪个队列 — 用于可观测性和调试
- 大 output（超过阈值）仍走文件存储 + `LogSliceRef`，`output` 字段只存 ref

##### 2.3 Router 和 Loop 处理

**Router**：Router step 完成后，编排者读取 Router step 的 output（包含匹配的 branch 信息），确定下一步是 branch 的第一个 step 还是 Router 之后的 step。

**Loop**：Loop step 作为特殊的 "子编排" — 编排者为每次迭代入队一组 step Job。Loop 内的 step 完成后，编排者检查是否还有下一次迭代，如果有则继续入队，否则跳到 Loop 之后的 step。

```
Router 完成 → 读取 output.branchIndex → 确定 branch → 入队 branch 的第一个 step
Loop 完成   → 读取 output.currentIteration → 还有下一次? → 入队循环体的第一个 step
           → 没有下一次? → 入队 Loop 之后的 step
```

##### 2.4 编排触发点

编排器在以下场景被触发：

1. **Trigger 执行完成** → 编排第一个 step
2. **Step 执行完成** → 编排下一个 step
3. **Waitpoint Resume** → 编排恢复后的 step
4. **Flow Run Retry** → 从失败的 step 重新开始

这些触发点在 `jobBroker.completeJob` 和 `resumeService` 中。

#### Phase 3: Step 执行器（Worker 端）

**目标**：Worker 能执行单个 Step，并从 Step Output Store 重建上下文。

##### 3.1 新增 `executeStepJob` handler

```typescript
// packages/server/worker/src/lib/execute/jobs/execute-step.ts

executeStepJob: JobHandler<ExecuteStepJobData, SynchronousJobResult> = {
    jobType: WorkerJobType.EXECUTE_STEP,
    async execute(ctx, data) {
        // 1. 获取 FlowVersion
        const flowVersion = await flowCache(ctx.log, ctx.apiClient).getVersion({
            flowVersionId: data.flowVersionId
        })

        // 2. 从 API 读取前序 steps output (Memoization)
        const stepOutputs = await ctx.apiClient.getStepOutputs({
            flowRunId: data.runId,
            projectId: data.projectId,
        })

        // 3. 重建 FlowExecutorContext
        const executionState = rebuildFlowExecutorContext(stepOutputs)

        // 4. 找到当前 step
        const step = flowStructureUtil.getActionOrThrow(
            data.stepName,
            flowVersion.trigger
        )

        // 5. 执行当前 step
        const handler = flowExecutor.getExecutorForAction(step.type)
        const newExecutionState = await handler.handle({
            action: step,
            executionState,
            constants: buildStepConstants(data, flowVersion),
        })

        // 6. 返回 step output
        return {
            kind: JobResultKind.SYNCHRONOUS,
            status: newExecutionState.verdict.status === FlowRunStatus.RUNNING
                ? EngineResponseStatus.OK
                : EngineResponseStatus.OK,  // 即使 FAILED 也返回 OK, 由编排者处理
            response: {
                stepOutput: newExecutionState.getStepOutput(data.stepName),
                verdict: newExecutionState.verdict,
                tags: Array.from(newExecutionState.tags),
            },
        }
    }
}
```

##### 3.2 FlowExecutorContext 重建

```typescript
// 从 Step Output Store 的数据重建 FlowExecutorContext
function rebuildFlowExecutorContext(
    stepOutputs: Record<string, StepOutput>
): FlowExecutorContext {
    return FlowExecutorContext.fromStepOutputs({
        steps: stepOutputs,
        engineApi: { engineToken, internalApiUrl },
    })
}
```

需要给 `FlowExecutorContext` 新增一个 `fromStepOutputs` 静态方法，从 `Record<string, StepOutput>` 直接构建实例。当前已有的 `empty()` 和从 logs file 恢复的路径可以参考。

##### 3.3 执行模式选择

在 `UNSANDBOXED` 模式下，step 执行直接在 Worker 进程内调用 Engine 函数，无需启动 Sandbox 子进程。这消除了每个 step 的进程启动开销（~1-3 秒）。

在其他模式下，可以复用现有 Sandbox 机制 — 但每个 step 需要独立的 sandbox 生命周期。`SandboxManager.acquire()` + `release()` 已支持此模式。

### 实施顺序与依赖关系

```
Phase 1 (Step Output Store)
   │
   ▼
Phase 2 (Step Orchestrator) ──▶ Phase 4 (Worker Group 配置)
   │
   ▼
Phase 3 (Step Executor)
   │
   ▼
Phase 5 (Feature Flag & 共存)
   │
   ▼
Phase 6 (可观测性)
```

- Phase 1 是所有后续阶段的基础
- Phase 2 和 Phase 3 可以并行开发，但需要 Phase 1 完成后才能集成测试
- Phase 4 是独立的功能扩展，可以在 Phase 2 之后任何时候加入
- Phase 5 是集成阶段，需要 Phase 2 和 Phase 3 都完成
- Phase 6 是增强阶段，可以在 Phase 5 之后逐步完善

### 风险与缓解

| 风险 | 严重度 | 缓解措施 |
|---|---|---|
| StepOutput 大对象存储到 JSONB 的性能问题 | 高 | 对大 output 走文件存储 + LogSliceRef，JSONB 只存 ref |
| Router 分支决策的重建逻辑复杂 | 高 | 先实现线性 step 编排，Router 作为 Phase 2.3 的子阶段 |
| Loop 迭代的 step 拆分导致 job 爆炸 | 中 | Loop 内部 step 仍作为一个 Job 组执行，不拆分每次迭代 |
| FlowExecutorContext 重建的正确性 | 高 | 逐字段对比测试：从 Step Output Store 重建的 context 与内存中的 context 必须一致 |
| 与现有 waitpoint 机制的兼容性 | 中 | waitpoint 仍作为 step 暂停的手段，只是恢复后入队的是 EXECUTE_STEP 而不是 EXECUTE_FLOW |
| 多 step 并发执行的竞态条件 | 中 | 同一个 Flow Run 的 step 串行执行，避免并发 |
| Step 间延迟增加（每个 step 需要一次 poll + 一次 API 调用） | 中 | UNSANDBOXED 模式 + step output 缓存优化 |

## 起手结论

- 项目类型：已有项目，架构改造
- 核心参照：Inngest 的 Memoization + Step Output Store 模式
- 最小可行路径：Phase 1 → Phase 2（线性 step 编排）→ Phase 3 → Phase 5
- 推荐入口：`feature-backend`，从 Phase 1 开始
- 最大风险：FlowExecutorContext 重建正确性 + Router/Loop 语义的编排逻辑复杂度
